This document is intentionally implementation-level. It is written to demonstrate product thinking, architecture decisions, execution depth, and delivery quality expected from a senior individual contributor.

1. Executive Summary

GPMS Web is a production operations cockpit for garment manufacturing. The frontend is not a static dashboard; it is an operational system where planners, supervisors, store teams, and QC teams continuously update state that directly impacts downstream workflow.

My implementation approach focused on three principles:

  • Workflow correctness first: UI actions are mapped to explicit backend transitions and guarded validations.
  • Single source of truth for context: factory scope and selected date are shared globally so all pages represent the same operational day/factory.
  • Low-friction operations: direct-entry forms, command palette, drawer details, and quick actions reduce navigation cost for floor users.

Key frontend outcomes delivered

  • PO lifecycle actions integrated in both table and detail drawer.
  • BOM editor upgraded from single-line submission to true multi-line authoring.
  • Date handling corrected to local-today default and synchronized across dashboard, sewing, QC, and topbar.
  • Dashboard CTA buttons wired to functional pages.
  • Linting infrastructure converted to non-interactive ESLint CLI for CI-friendly quality gates.
  • Interview-ready and engineering-ready documentation added.

2. Product Context and End-to-End Flow

The web app models a realistic factory process: 

Buyer -> Style + BOM -> PO -> Material Requirement -> Inventory -> Planning -> Cutting -> Sewing -> QC -> Shipment/Closure

The frontend supports each stage with dedicated route modules and shared context controls (factory + date + search).

3. Frontend Architecture

3.1 Application Shell

  • Root layout: font loading, MUI cache provider, global providers.
  • Protected layout: all operational pages wrapped in auth gate.
  • App shell: persistent sidebar, topbar, command palette, contextual action drawer.

3.2 Provider Stack

Global providers are initialized once and used across routes:

  • QueryClientProvider for server state and caching.
  • ThemeProvider (MUI dark-first theme with custom design tokens).
  • AuthProvider for session hydration and user/factory bootstrap.
  • Toaster for action feedback and error visibility.

3.3 Auth and Access Flow

  • Session hydration on startup with GET /auth/me.
  • If unauthenticated, immediate redirect to /login?next=....
  • Factory context auto-selected from persisted value, activeFactory, or first accessible factory.
  • Logout performs backend logout and client state cleanup.

3.4 Global State (Zustand)

State model balances persistence and correctness:

  • Persisted: factoryId, theme, sidebar state.
  • Not persisted by design: selectedDate (resets to local today each load).
  • Operational utilities: command palette state, action intents, global search.
Key design decision: avoid persisting date to prevent stale operational context on next-day logins.

3.5 API Integration Contract

  • Axios client with credentials enabled by default.
  • Request interceptor injects x-factory-id.
  • Defensive request normalization for plan assignment payloads.
  • All user-facing mutations report success/failure via toast notifications.

4. Navigation and Usability Design

  • Sidebar navigation for full module coverage.
  • Topbar as universal context bar (factory, date, global search).
  • Command palette (Cmd/Ctrl + K) for quick jumps and guided action intents.
  • Right-side drawer pattern for details to reduce page transitions.
  • Live dashboard CTAs for direct operational routing.

5. Module-Level Process Walkthrough

5.1 Buyers

CRUD lifecycle with edit/delete actions in detail drawer. Supports buyer master reliability before PO creation.

5.2 Styles and BOM

This module is foundational because PO material requirement quality depends on BOM quality.

  • Style create/update/delete supported.
  • Multi-line BOM editing implemented in frontend.
  • Duplicate materials prevented client-side before save.
  • Saves full items[] payload in one operation (compatible with backend replace semantics).

5.3 Materials

Material master setup (name, type, UOM). Used by BOM, inventory ledger, and PO requirement views.

5.4 Purchase Orders

PO screen includes create, add items, status progression, requirement visibility, and stock coverage.

  • Actions in table and detail drawer.
  • Status controls aligned to backend guards.
  • Material requirement CSV export available for operational handoff.

5.5 Inventory

Supports both inbound receive and issue-to-cutting operations, with running stock position table.

5.6 Planning

Planner can create plan, assign PO item to sewing line, and define dated targets.

Assignment integrates downstream status behavior (PO enters production when assignment begins).

5.7 Cutting

  • Create cutting batch against PO item.
  • Generate bundles with size/qty lines.
  • Bundle state and detail exploration through table + drawer.

5.8 Sewing

  • Submit hourly output by line/hour/qty and optional bundle linkage.
  • Live line status polling and WIP visibility.
  • Date control synchronized to global selected date (fixed drift issue).

5.9 QC

  • Inspection entry (INLINE/FINAL, pass/fail, defects).
  • QC summary by selected date.
  • Date also synchronized to global selected date.

5.10 Dashboard

  • Aggregates plan-vs-actual, line status, and QC summary.
  • Includes quick hourly output posting with optimistic updates.
  • Live status strip with incremental seconds since last update.
  • CTAs route directly to PO, Sewing, and QC operational pages.

6. Engineering Decisions and Tradeoffs

6.1 Why React Query for server state

  • Built-in stale/refetch lifecycle for manufacturing dashboards.
  • Easy query invalidation after mutations.
  • Supports optimistic patching where immediate feedback matters.

6.2 Why keep factory/date in shared store

  • Avoids inconsistent cross-page context.
  • Prevents accidental queries against wrong factory/day.
  • Reduces user mistakes during rapid operational switching.

6.3 Status progression discipline

Frontend does not fake lifecycle completion. Status buttons call backend transitions that enforce operational readiness checks. This keeps UI behavior trustworthy under real usage.

6.4 UI system strategy

  • MUI for robust controls and behavior.
  • Tailwind + utility wrappers for layout speed and visual consistency.
  • Reusable app shell and drawer model to standardize flows.

7. Reliability, Validation, and Error Handling

7.1 Input validation

Centralized utility checks for required text, valid dates, positive quantities, and date order constraints.

7.2 Mutation feedback pattern

  • All mutations show success/failure toasts.
  • Error messages prioritize backend response payload if present.
  • Critical lists/details are invalidated after successful mutations.

7.3 Polling and freshness

  • Dashboard and line status poll periodically where data volatility is high.
  • Incremental update counter gives operators confidence in recency.

7.4 Lint and build posture

  • ESLint CLI now non-interactive and CI-compatible.
  • Build pipeline passes consistently.
  • Known warnings remain and are explicitly trackable for debt cleanup.

The implementation choices prioritize real factory usability over demo-only polish, while still maintaining scalable architecture and clean extensibility.
